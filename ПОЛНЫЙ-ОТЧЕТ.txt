1. ЦЕЛЬ ПРОЕКТА
--------------------------------------------------------------------------------
  • Удалённый доступ к серверу по SSH
  • Установка и работа мощных текстовых и vision-моделей через Ollama
  • Удобный доступ через Telegram-бота
  • Русский язык, память диалога, режим «рассуждения» (явный CoT в ответе)


2. АППАРАТНАЯ ЧАСТЬ (ИСХОДНЫЕ ДАННЫЕ)
--------------------------------------------------------------------------------
  Сервер: Dell PowerEdge R760
  CPU:     2× Intel Xeon Silver 4516Y+
  RAM:     512 GB DDR5
  GPU:     NVIDIA A40, 48 GB VRAM
  Диски:   3× 960 GB NVMe, 5× 1.8 TB SAS (по факту использованы 2 NVMe под RAID)
  Сеть:    2× 10GbE SFP+
  Удалённое управление: iDRAC9 Enterprise


3. ОПЕРАЦИОННАЯ СИСТЕМА И СЕТЬ
--------------------------------------------------------------------------------
  ОС на сервере: Ubuntu Server 24.04 LTS (ядро 6.8.x)
  Локальный IP сервера в LAN: YOUR_LAN_HOST
  Шлюз: 192.168.1.1
  Внешний доступ (пример): YOUR_PUBLIC_HOST
  Проброс портов на роутере: внешний порт 42 → YOUR_LAN_HOST:22 (SSH)

  Важно: с «чужой» сети доступ к 192.168.x.x невозможен — использовался внешний IP
  и проброс порта 42 для SSH.


4. ЧТО СДЕЛАНО НА СЕРВЕРЕ (LINUX)
--------------------------------------------------------------------------------
  4.1 Обновления и базовые инструменты
      • apt update / установка пакетов (в т.ч. для работы с Python venv при необходимости)

  4.2 NVIDIA
      • Установлены проприетарные драйверы NVIDIA (в процессе были конфликты пакетов
        dpkg — выполнялось принудительное удаление конфликтующих пакетов и
        повторная установка драйверов, после перезагрузки — nvidia-smi показывает A40)

  4.3 Docker и NVIDIA Container Toolkit
      • По плану проекта предусматривались Docker + nvidia-container-toolkit
        (для альтернативных сценариев vLLM/TGI; основной путь — Ollama)

  4.4 Ollama
      • Установлен как основной движок для LLM
      • Сервис ollama включён и слушает API (по умолчанию localhost:11434, при
        необходимости — настройка на 0.0.0.0 для доступа из сети)
      • Загружены модели (см. раздел 6)

  4.5 Дисковое пространство и RAID
      Проблема: корневой том был заполнен (~100%), загрузка крупных моделей
      останавливалась с ошибкой «no space left on device».

      Решение:
      • Создан программный RAID0 (mdadm) из двух свободных NVMe: nvme1n1 + nvme2n1
      • Устройство: /dev/md0
      • Файловая система: ext4
      • Точка монтирования: /mnt/models
      • Каталог данных Ollama перенесён на этот том, на старом месте — симлинк
      • Запись в /etc/fstab для автомонтирования после перезагрузки
      • Права: пользователь ollama, владелец каталога моделей

  4.6 Файрвол
      • UFW по диагностике был неактивен на момент проверки; правила под SSH/Ollama
        обсуждались в документации (на сервере при необходимости уточнять ufw status)

  4.7 Кастомная модель qwen-pro
      • На базе qwen2.5:72b создана модель Ollama с именем qwen-pro
      • В Modelfile заданы параметры (temperature, top_p, num_ctx и т.д.) и SYSTEM
        с жёсткими правилами: русский язык, JSON при запросе, пункты, честность
        при отсутствии данных
      • Примечание: при копировании через heredoc часть текста могла обрезаться;
        при странном поведении модели стоит пересоздать Modelfile целиком и
        выполнить ollama create заново


5. ДОСТУП К СЕРВЕРУ С WINDOWS
--------------------------------------------------------------------------------
  • SSH: ssh -p 42 user@YOUR_PUBLIC_HOST (или другой внешний IP / порт по факту)
  • Для локального бота на ПК: туннель Ollama
      ssh -p 42 -L 11434:localhost:11434 user@<IP>
    Тогда на ПК запросы идут на http://localhost:11434 и попадают на серверный Ollama


6. МОДЕЛИ OLLAMA (ПО СОСТОЯНИЮ РАБОТЫ)
--------------------------------------------------------------------------------
  Загружались и использовались (имена могут совпадать с ollama list на сервере):

  • qwen2.5:72b       — основной текст (~47 GB)
  • qwen-pro          — кастомная обёртка над qwen2.5:72b с SYSTEM-промптом
  • deepseek-coder:33b — код (~18 GB)
  • llama3.2-vision:90b — vision (~54 GB)

  Дополнительно в документации упоминались варианты qwen-ru, coder-ru, vision-ru
  и скрипты оптимизации — фактическое наличие зависит от того, что выполнено на сервере.


7. TELEGRAM-БОТ
--------------------------------------------------------------------------------
  7.1 Почему бот на Windows, а не на сервере
      На сервере при проверке curl к api.telegram.org:443 соединение устанавливалось,
      но TLS «зависал» — типичная картина блокировки/фильтрации Telegram API.
      Поэтому рабочая схема: бот запускается на ПК пользователя, Telegram доступен
      с локальной машины; к Ollama — через SSH-туннель на порт 11434.

  7.2 Эволюция кода
      • Ранние версии: telegram-bot.py, telegram-bot-local.py
      • Актуальная расширенная версия: telegram-bot-advanced.py
        - SQLite: чаты, сообщения, активный чат, настройки пользователя
        - Режим «рассуждение»: промпт с блоками РАССУЖДЕНИЕ / ОТВЕТ, разбор ответа
          на два сообщения в Telegram (это не скрытый «мозг» модели, а явный текст
          по инструкции)
        - Команды: /start, /chats, /reason и др.
      • start-all.ps1 — сценарий запуска (обновлялся под advanced-бот)

  7.3 Функции бота (итог)
      • Инлайн-кнопки (меню, чаты, помощь, переключение режима рассуждений)
      • Диалоги с сохранением в БД (bot_chats.db рядом с скриптом на ПК)
      • Приоритет модели: qwen-pro → qwen-ru → qwen2.5:72b (если имена есть в ollama list)
      • Усиление ответов на русском через промпты / Modelfile
      • Документация: тестовые вопросы, запуск, память, система чатов

9. ИЗВЕСТНЫЕ ОГРАНИЧЕНИЯ И РИСКИ
--------------------------------------------------------------------------------
  • Токен Telegram бота многократно фигурировал в чате и в файлах — его нужно
    считать скомпрометированным: в BotFather отозвать и выдать новый, обновить
    в коде и нигде не публиковать.

  • «Автообучение» от каждого сообщения не реализовано: веса модели не меняются
    от диалогов. Улучшение ощущений даёт память чата и промпты, не дообучение.

  • RAID0: выше скорость/объём, но отказ одного диска теряет массив — нужен бэкап
    важных данных.

  • Бот на ПК: пока запущен туннель и бот — работает; при выключении ПК бот недоступен
    (для 24/7 — перенос бота на машину с доступом к Telegram или VPN/прокси на сервере).
